# Project Section

## Purpose

Use projects to prove relevant skills when work experience alone does not show enough evidence for the target JD.

## What Good Looks Like

Projects should be selected for relevance, credibility, and signal. A strong project entry explains problem, users/data/scope, technology, candidate role, and outcome.

## Weak vs Strong

Weak: Todo app using React.

Strong: Built a React/Node scheduling app with PostgreSQL auth and calendar sync; deployed demo on Vercel and used it with 12 beta users to validate reminder flows.

Weak: Machine learning project.

Strong: Trained and evaluated a churn model on 120k subscription events, improving recall from 0.61 to 0.74 after feature engineering and threshold tuning.

## Role-Specific Notes

- Junior engineers: projects can sit above experience if they are more relevant than internships.
- Data/AI: include dataset, method, evaluation metric, and deployment/dashboard if any.
- Product: frame case studies around customer problem, decision, trade-off, and result.
- Senior candidates: include projects only if open-source, portfolio, or strategically relevant.

## Common Mistakes

- Listing too many toy projects.
- Omitting GitHub/demo links when the project is public.
- Describing tools but not problem/outcome.
- Including abandoned experiments that cannot be explained.

## Source Patterns

Coursera and Resume Worded both recommend projects for newer or transitioning candidates when they show applied work. See `../sources.md`.
